# 用户角色 {#user-roles}

体验转出提供五个角色。 每个角色都针对一组特定的责任而设计。 默认情况下，角色是互斥的 — 具有管理员角色的成员不会自动拥有产品版本所有者权限。 需要更广泛的访问权限时，为成员分配多个角色。

## 可用的角色 {#roles}

| 角色 | 责任 |
|---|---|
| **管理员** | 将应用程序置于控制台，管理团队成员，以及批准或拒绝访问请求。 在团队可以开始使用功能标记之前需要。 |
| **产品版本所有者** | 为其应用程序创建和更新功能标记和功能组。 可将受众链接到功能标记，以使功能可用于外部用户。 |
| **开发人员** | 在沙盒上下文中工作，以测试功能标记后的功能，而不影响其他用户。 可以在暂存或生产环境中向专用受众（例如特定用户ID或电子邮件地址）公开功能标记。 |
| **功能管理员** | 创建和管理跨团队功能组。 在协调不同团队拥有的应用程序中的标记时，使用此角色。 |
| **发行经理** | 管理多团队、多应用程序版本并更新定义接收版本的用户定义的受众规则。 |

## 选择正确的角色 {#choosing}

使用以下指南分配角色：

* **正在为您的应用程序添加或更新功能标志** → **产品版本所有者**
* **在不影响其他人的情况下单独测试功能** → **开发人员**
* **管理跨团队功能组** → **功能管理员**
* **跨多个团队和应用程序管理版本** → **版本管理器**
* **登录应用程序并管理团队成员资格** → **管理员**

>[!NOTE]
>
>一个成员可以担任多个角色。 例如，同时测试功能和管理团队应用程序的工程师应同时具有&#x200B;**开发人员**&#x200B;和&#x200B;**管理员**&#x200B;角色。

## 权限错误 {#permissions-error}

如果成员在尝试更新功能或组时遇到“权限不足”错误，则需要&#x200B;**产品版本所有者**&#x200B;角色。 请联系您的团队管理员以添加此角色。
